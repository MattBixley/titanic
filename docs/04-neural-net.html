A small multilayer perceptron with nnet — and why scale matters

Where we are: Last of three model documents in the Titanic — Tidymodels Tour. Same structure as the XGBoost and Random Forest docs — recipe → spec → workflow → tune → finalize → evaluate → save. The one twist: a neural net needs normalized inputs, so this doc extends the shared make_recipe() with a normalization step.

Why a neural net here?

On a dataset this small (891 rows) a multilayer perceptron will not beat the tree ensembles — and that’s fine. The point is to explore a different model family and see how tidymodels makes swapping engines almost free. A small MLP with a single hidden layer and weight decay (penalty) is a clean, instructive choice.

NoteEngine choice: nnet

We fit the MLP with base R’s nnet engine. It has no native-code dependency, so it trains identically on your laptop, in the Quarto site build, and in GitHub Actions CI. A torch-backed alternative (brulee) is shown later as an optional local upgrade — but it does not load inside this project’s render/CI session, so nnet is the engine we publish.

The recipe — with normalization

Tree ensembles are invariant to monotone rescaling, so the shared recipe didn’t bother normalizing. A neural net is not invariant: gradient descent behaves far better when every numeric predictor has mean 0 and unit variance, otherwise large-scale features (like Fare) dominate the gradient. So we take the shared recipe and append step_normalize().

Show code 
# Extend the shared recipe with normalization of all numeric predictors. This
# step must come AFTER dummy encoding in make_recipe() so the 0/1 dummy columns
# are normalized too (standard practice for MLPs).
nnet_recipe <- make_recipe(train)

The model specification

Show code 
# Single-hidden-layer MLP via base R's nnet engine: one hidden layer
# (hidden_units) and L2 weight decay (penalty). nnet has no native-code
# dependency, so it trains and renders reliably everywhere — including the
# Quarto site build and GitHub Actions CI.
nnet_spec <- mlp(
  hidden_units = tune(),
  penalty      = tune()
) |>
  set_engine("nnet") |>
  set_mode("classification")

nnet_spec
Single Layer Neural Network Model Specification (classification)

Main Arguments:
  hidden_units = tune()
  penalty = tune()

Computational engine: nnet
TipOptional: a torch-backed MLP with brulee (local only)

For a deeper net with a tuned epoch budget you can swap in brulee (a torch backend) when running locally. It also exposes epochs:

nnet_spec <- mlp(
  hidden_units = tune(),
  penalty      = tune(),
  epochs       = tune()       # brulee only
) |>
  set_engine("brulee") |>
  set_mode("classification")

We keep nnet as the published engine on purpose: brulee’s libtorch backend does not load inside the Quarto/knitr render session (or in CI) under this project’s renv setup — every fit fails with “Lantern is not loaded”. nnet sidesteps that entirely. If you switch to brulee, add epochs back to the tuning grid below and train sequentially (foreach::registerDoSEQ()).

# The recipe now adds PCA internally, so num_comp is a tunable parameter too.
# Bake a small recipe to give finalize() the predictor columns it needs to set
# sensible ranges (e.g. hidden_units).
preds <- prep(make_recipe(train, num_comp = 2)) |>

nnet_params <- extract_parameter_set_dials(nnet_wf) |>

Variable importance

Neural nets don’t expose importance natively. For an nnet model vip derives it from the trained connection weights (Garson’s algorithm) — no shuffling or prediction wrapper needed, so it’s robust in any render environment.

Show code 
# Pull the underlying nnet engine object out of the fitted workflow and let vip
# read importance straight from the network weights.
nnet_fit |>
  extract_fit_engine() |>
  vip(num_features = 15) +
  labs(title = "Neural Net variable importance (Garson weights)") +
  theme_minimal()

Even through a different lens, the same features carry the weight — Title, Sex, class, Fare — consistent with XGBoost and the Random Forest.

## Tuning with 5-fold cross-validation

```{r}
#| label: resamples
set.seed(42)
folds <- vfold_cv(train, v = 10, strata = Survived)
```

```{r}
#| label: grid
# The recipe now adds PCA internally, so num_comp is a tunable parameter too.
# Bake a small recipe to give finalize() the predictor columns it needs to set
# sensible ranges (e.g. hidden_units).
preds <- prep(make_recipe(train, num_comp = 2)) |>
  bake(new_data = NULL) |>
  select(-Survived, -PassengerId)

nnet_params <- extract_parameter_set_dials(nnet_wf) |>
  finalize(preds) |>
  update(num_comp = num_comp(range = c(2L, 10L)))

set.seed(42)
nnet_grid <- grid_space_filling(nnet_params, size = 18)
```

```{r}
#| label: tune
#| message: false
#| warning: false

nnet_metrics <- metric_set(roc_auc, accuracy)

set.seed(42)
nnet_tuned <- tune_grid(
  nnet_wf,
  resamples = folds,
  grid      = nnet_grid,
  metrics   = nnet_metrics,
  # Positive class "1" is the second factor level.
  control   = control_grid(save_pred = TRUE, save_workflow = TRUE, event_level = "second")
)
```

```{r}
#| label: save-res
saveRDS(nnet_tuned, here("models", "nnet_res.rds"))
```
